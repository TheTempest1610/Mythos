## Asset Types

| Asset | Storage | Notes |
| --- | --- | --- |
| Entity sprites | `Resources/Textures/_Mythos/**/*.rsi/` | RSI folders contain `meta.json` and PNG state sheets. Use from `Sprite` / `Icon` components. |
| Tile sprites | `Resources/Textures/Tiles/_Mythos/**/*.png` | Tiles use direct PNG paths, not RSI state lookup. |
| Tile sidecars | `Resources/Textures/Tiles/_Mythos/**/_manifest.json` | Generated by `mktiles.py`; records source DMI/state and variants for diffing. |
| Tile prototypes | `Resources/Prototypes/_Mythos/Tiles/**/*.yml` | `type: tile`, `id`, `name`, `sprite`, optional `variants`, `edgeSprites`, physics/collision fields. |
| Tile locale | `Resources/Locale/en-US/_Mythos/tiles/ov-tiles.ftl` | Generated seed strings need human cleanup over time. |
| Maps | `Resources/Maps/*.yml` | Serialized map files. |
| Map prototypes | `Resources/Prototypes/Maps/*.yml` | Selectable game map metadata and station/settlement configs. |

## OV Tile Import Pipeline

The current imported fantasy terrain comes from `Tools/_Mythos/ov_port/`.

Typical flow:

```powershell
python Tools\_Mythos\ov_port\mktiles.py <dmi_or_dir> Resources\Textures\Tiles\_Mythos\_OV Resources\Prototypes\_Mythos\Tiles\_OV
python Tools\_Mythos\ov_port\seed_locale.py --repo .
python Tools\_Mythos\ov_port\validate_tile.py --repo . --batch _OV
python Tools\_Mythos\ov_port\diff_tiles.py Resources\Textures\Tiles\_Mythos\_OV
```

`mktiles.py` emits:

- flat tile PNGs under `Resources/Textures/Tiles/_Mythos/_OV/<category>/`
- `_manifest.json` sidecars under each category
- tile prototype YAML under `Resources/Prototypes/_Mythos/Tiles/_OV/`

Tile prototype example:

```yaml
- type: tile
  id: FloorMythosRoguefloorWoodenFloor
  name: tiles-ov-roguefloor-wooden_floor
  sprite: /Textures/Tiles/_Mythos/_OV/roguefloor/wooden_floor.png
  variants: 2
  placementVariants:
  - 0
  - 1
```

Agents should not manually edit generated tile YAML unless the change is deliberate and cannot be represented in the importer. Generated files begin with:

```text
# Auto-generated by Tools/_Mythos/ov_port/mktiles.py
```

## Variant-Aware Tile Spawn

Stock tile placement always writes variant `0`. Mythos adds a mapper UI to choose a specific variant.

Flow:

1. `SandboxUIController` uses `MythosTileSpawningUIController` instead of the stock tile spawner.
2. `MythosTileItemList` detects which 32px variant cell was clicked in the tile icon strip.
3. Client begins tile placement and sends `MsgSetTileVariantOverride(TileType, Variant)`.
4. Server `TileVariantOverrideSystem` stores override per `NetUserId`.
5. On `PlacementTileEvent`, the server checks Mapping admin permission and matching tile type.
6. Server re-writes the placed tile with the chosen variant.
7. Variant `0` clears the override.

Search targets:

```powershell
rg -n "MythosTileSpawningUIController|MythosTileItemList|MsgSetTileVariantOverride|TileVariantOverrideSystem|PlacementTileEvent" Content.Client\_Mythos Content.Server\_Mythos Content.Shared\_Mythos Content.Client\UserInterface\Systems\Sandbox
```

## Mapping Workflow

For Mythos maps, keep the SS14 command workflow but ignore station-specific content unless a system still needs the internal component.

Useful commands:

```text
mapping 1000
mapping 1000 Maps/example.yml
mappingclientsidesetup
lsmaps
lsgrid
savemap 1000 mythos_example.yml
forcemap MythosExample
restartroundnow
togglelight
zoom 3
```

When creating a playable map:

1. Save the map into `bin/Content.Server/data/`.
2. Move the stable copy into `Resources/Maps/`.
3. Add or update a `gameMap` prototype under `Resources/Prototypes/Maps/`.
4. Add the map to the relevant `gameMapPool`.
5. Set local defaults in `config/server_config.toml` only for development.
6. Test with `forcemap <map prototype id>` and start/restart the round.

## Station Internals in a Fantasy Map

Many round systems still expect a station entity and station config. For now, treat "station" as an internal owner entity that can represent a settlement, dungeon, keep, or realm.

Keep these internals until replacements are designed:

- `BecomesStation`
- station config in `gameMap` prototypes
- job availability under station config
- station spawning markers
- station records and announcements if referenced by active systems

Player-facing names should say settlement, realm, guild hall, dungeon, or keep through locale/prototypes/UI. Internal component names can remain until a deliberate round-flow rewrite.

## Sprite and RSI Rules

Entity sprites should follow Robust Station Image rules:

- folder name ends in `.rsi`
- `meta.json` defines states
- state PNG names match the state IDs
- prototypes reference the RSI folder and state

Example:

```yaml
- type: Sprite
  sprite: _Mythos/_OV/Roguetown/Weapons/swords32.rsi
  state: longsword
- type: Icon
  sprite: _Mythos/_OV/Roguetown/Weapons/swords32.rsi
  state: longsword
```

Tiles use direct texture paths:

```yaml
sprite: /Textures/Tiles/_Mythos/_OV/roguefloor/cobblestone.png
```

## Localization Rules

Do not hardcode player-facing strings in C# unless the surrounding code already does and no localization path exists. Prefer Fluent keys:

```csharp
Loc.GetString("mythos-spell-no-mana", ("spell", spellName))
```

```fluent
mythos-spell-no-mana = You do not have enough mana to cast {$spell}.
```

Generated tile locale is intentionally rough. Improve display strings in `Resources/Locale/en-US/_Mythos/tiles/ov-tiles.ftl` without changing tile IDs.
